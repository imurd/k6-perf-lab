# Мок-интервью Performance-инженера (тренажёр)

Файл для самостоятельной отработки под запись. Формат каждого вопроса:

```
❓ ВОПРОС
⏸  Останови чтение. Вслух ответь сам (30–60 сек). Только потом читай дальше.
✅ ЭТАЛОН — минимум, который ждут (не заучивай дословно, говори своими словами).
👀 НА ЧТО СМОТРИТ ИНТЕРВЬЮЕР / 🚩 красные флаги.
```

## Как пользоваться
1. Включи диктофон/запись экрана.
2. Иди по сессии подряд, не подсматривая в эталон до своего ответа.
3. После сессии прослушай себя: где «э-э-э», где расплывчато, где не назвал цифры.
4. Оценка по рубрике внизу. Повторяй сессию, пока не выйдешь на «уверенно» по всем.

Три сессии по возрастанию сложности + раунд behavioral + практический раунд.

# СЕССИЯ A — Скрининг (Junior уровень, фундамент)

### A1
❓ Чем latency отличается от throughput?
⏸
✅ Latency — время ответа на один запрос (мс). Throughput — сколько запросов в секунду обрабатывается (rps). Это независимые вещи: бывает высокий throughput при плохой latency.
👀 Смотрят: не путаешь ли понятия. 🚩 Красный флаг: «это одно и то же».

### A2
❓ Почему при анализе смотрят на p95/p99, а не на среднее время ответа?
⏸
✅ Среднее прячет «хвост». 1% очень медленных запросов почти не двигает среднее, но это реальные страдающие пользователи. p99 = «99% быстрее X» — честно показывает худший опыт.
👀 Понимание перцентилей. 🚩 «Среднее и так всё показывает».

### A3
❓ Что такое SLO и чем отличается от SLA?
⏸
✅ SLO — внутренняя цель по метрике (p95 < 300 мс). SLA — внешний контракт с клиентом и санкциями. SLI — сам измеряемый показатель. SLO обычно строже SLA.
👀 Знание терминологии надёжности. 🚩 Не различает внутреннее/внешнее.

### A4
❓ Что такое виртуальный пользователь (VU) в нагрузочном тесте?
⏸
✅ Параллельный «поток-робот», который в цикле выполняет сценарий. Число VU = уровень одновременной нагрузки.
👀 Базовое понимание инструмента. 🚩 Путает VU с числом запросов.

### A5
❓ Назови основные типы нагрузочных тестов.
⏸
✅ Load (ожидаемая нагрузка), Stress (выше нормы, ищем потолок), Soak/endurance (долго на средней — утечки), Spike (резкий скачок). Иногда Volume (объём данных).
👀 Кругозор по методологии. 🚩 Знает только «нагрузить и посмотреть».

### A6
❓ Зачем в сценарии нужны паузы (think time / sleep)?
⏸
✅ Имитируют поведение реального пользователя, который читает страницу. Без пауз поток нереалистичен и нагрузка искусственно завышена.
👀 Реалистичность моделирования. 🚩 «Паузы не нужны, надо жать максимум».

# СЕССИЯ B — Технический раунд (Middle уровень)

### B1
❓ Что такое методология USE и как ей искать узкое место?
⏸
✅ Для каждого ресурса (CPU, память, диск, сеть) смотрим Utilization (загрузка), Saturation (очередь/ожидание), Errors. Насыщенный ресурс с очередью — кандидат в bottleneck. Быстрый способ не гадать. (Gregg.)
👀 Системный подход к диагностике. 🚩 «Смотрю на CPU и всё».

### B2
❓ Latency выросла под нагрузкой. Твой алгоритм поиска причины?
⏸
✅ 1) В k6 разложить время по фазам (connect/TLS/waiting=TTFB) — где теряется. 2) На сервере по USE: CPU (`top`), память/swap (`vmstat`), диск (`iostat`), сеть. 3) Приложение/БД: медленные запросы, индексы, размер пула соединений, N+1, отсутствие кэша. Двигаюсь от измерения к гипотезе, а не наугад.
👀 Структурность. 🚩 Хаотичное «ну добавлю серверов».

### B3
❓ Разница между checks и thresholds в k6?
⏸
✅ checks — проверки каждого ответа (статус, тело), копят статистику, но не роняют прогон. thresholds — критерии прохождения всего теста (p95<500, ошибок<1%), их провал даёт exit code 1 и ломает CI.
👀 Понимание автоматизации. 🚩 Считает их одним и тем же.

### B4
❓ Что происходит с системой при приближении загрузки к 100%? Почему?
⏸
✅ Латентность растёт нелинейно (лавинообразно) из-за роста очередей — следствие теории очередей/закона Литтла. Поэтому целятся не в 100%, а в ~70–80% утилизации с запасом.
👀 Понимание нелинейности. 🚩 «Загрузим на 100%, это же эффективно».

### B5
❓ Как встроить нагрузочный тест в CI/CD и зачем?
⏸
✅ Прогон k6 в пайплайне с thresholds по SLO; провал порога → красный билд. Ловим регрессию производительности до прода. Тест — короткий (smoke/load), полноценный stress — отдельно.
👀 Инженерная зрелость. 🚩 Не понимает, зачем автоматизировать.

### B6
❓ Из каких фаз состоит время HTTP-запроса?
⏸
✅ DNS → TCP handshake → TLS handshake → отправка → ожидание сервера (TTFB) → приём тела. В k6: connecting, tls_handshaking, sending, waiting, receiving.
👀 Понимание сети. 🚩 «Просто время ответа, без деталей».

### B7 (мини-практика вслух)
❓ У тебя p95=1200мс при 200 VU, error rate 0%. CPU сервера 40%, диск загружен на 95% с большой очередью. Твой вывод?
⏸
✅ Bottleneck — диск (высокая утилизация + saturation по очереди), CPU свободен. Гипотезы: тяжёлые дисковые операции/логи/своп/неоптимальные запросы к БД, бьющие в диск. Дальше — смотреть iostat await, медленные запросы БД, индексы.
👀 Умение читать метрики вместе. 🚩 «CPU 40%, значит всё ок».

# СЕССИЯ C — Senior раунд (архитектура, компромиссы, инциденты)

### C1
❓ Как спланировать нагрузочное тестирование для нового сервиса с нуля?
⏸
✅ Собрать требования (ожидаемый трафик, пиковые сценарии), определить SLO, построить профиль нагрузки из реальных данных (логи прода/аналитика), выбрать инструмент, начать со smoke→load→stress→soak, снимать серверные метрики, зафиксировать baseline, автоматизировать в CI, документировать потолок и рекомендации по ёмкости.
👀 Видение процесса целиком. 🚩 Сразу лезет писать скрипт без целей.

### C2
❓ Circuit breaker и bulkhead — что решают и когда применять?
⏸
✅ Circuit breaker размыкается при стабильных отказах зависимости и быстро отдаёт ошибку/заглушку, давая ей восстановиться — против каскадных отказов. Bulkhead изолирует ресурсы по «отсекам» (отдельные пулы), чтобы отказ одной функции не утопил остальные. (Nygard.)
👀 Знание паттернов устойчивости. 🚩 Знает названия, но не «зачем».

### C3
❓ Система «в среднем держит», но падает на пике. Как объяснишь и что предложишь?
⏸
✅ Нелинейность очередей + каскады: при насыщении растут таймауты, ретраи усиливают нагрузку → лавина. Предложу: stress/spike-тесты для воспроизведения, лимиты и backpressure, circuit breaker, авто-масштабирование, деградация функционала под пиком, запас по утилизации.
👀 Системное мышление. 🚩 «Просто добавить памяти».

### C4
❓ Как из результатов тестов планировать ёмкость (capacity planning)?
⏸
✅ Из stress-теста — потолок (rps на инстанс). Из прод-трендов — прогноз роста. Считаю нужное число инстансов с запасом на пик и на отказ части узлов, закладываю сроки закупки/масштабирования. (Allspaw.)
👀 Связь тестов с бизнесом. 🚩 Не умеет перевести тест в «сколько железа».

### C5
❓ Горизонтальное vs вертикальное масштабирование — компромиссы?
⏸
✅ Вертикальное — проще, но есть потолок и единая точка отказа. Горизонтальное — сложнее (балансировка, состояние, согласованность), но масштабируется и надёжнее. Выбор зависит от stateful/stateless и стоимости.
👀 Умение обсуждать trade-offs. 🚩 «Горизонтальное всегда лучше».

### C6
❓ Расскажи про самый сложный перф-инцидент/оптимизацию из практики.
⏸
✅ (Твоя реальная история по схеме: контекст → симптом в цифрах → как измерял → корневая причина → решение → результат «было→стало» → что вынес.) Обязательно с цифрами (p95 X→Z, rps Y→W).
👀 Глубина реального опыта. 🚩 Общие слова без цифр и без корневой причины.

# ПРАКТИЧЕСКИЙ РАУНД (дают за компьютером)

### П1 — Написать сценарий
❓ Напиши k6-сценарий: разгон до 50 VU за 1 мин, плато 3 мин, спад 1 мин, порог p95<400мс и ошибок<1%.
⏸
✅ `options.stages = [{duration:'1m',target:50},{duration:'3m',target:50},{duration:'1m',target:0}]`, `options.thresholds = {http_req_duration:['p(95)<400'], http_req_failed:['rate<0.01']}`, в default — `http.get` + `check` + `sleep`. (См. `02-load.js` как образец.)

### П2 — Прочитать вывод
❓ Дают summary: `http_req_duration p(95)=850ms`, `http_req_failed 3.2%`, порог был `p(95)<500, rate<0.01`. Вердикт?
⏸
✅ Тест провален по обоим порогам: p95 в 1.7 раза выше цели и ошибок втрое больше допустимого. Дальше — искать причину по фазам запроса и USE на сервере.

### П3 — Спроектировать
❓ Как протестируешь эндпоинт логина, где нужен токен для последующих запросов?
⏸
✅ В сценарии: сначала POST /login, достать токен из ответа (`res.json('token')`), дальше слать запросы с заголовком `Authorization: Bearer <token>`. Логин-запрос учитывать отдельно, чтобы его латентность не смешивалась с остальными (группы/теги).

## РУБРИКА САМООЦЕНКИ (после каждой сессии)

| Критерий | 🔴 Слабо | 🟡 Норм | 🟢 Уверенно |
|---|---|---|---|
| Правильность | Ошибки по сути | В целом верно | Точно + нюансы |
| Структура ответа | Хаотично | Есть логика | Чёткий алгоритм |
| Цифры/примеры | Нет | Иногда | Всегда конкретно |
| Речь | Много «э-э», паузы | Сносно | Гладко, по делу |
| Время ответа | >2 мин или 5 сек | ~1 мин | 30–90 сек по делу |

**Готов к собеседованию**, когда все сессии — 🟢, и практический раунд делаешь без подсказок.
